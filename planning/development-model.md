# 软件工程开发模型

> 适用项目：flstudio-mcp-plus（FL Studio MCP Server）
> 模型来源：综合 IEEE/ISO 标准 + Agile 实践，针对个人项目精简

## 模型总览

```
基础诉求 → 调研 → 需求定义 → 架构设计 → 技术原型 → 项目计划 → 迭代实施
    ↑                                                          │
    └──────────────── 反馈循环 ←────────────────────────────────┘
```

---

## 阶段一：基础诉求（Problem Definition）

**目标：** 明确"为什么要做"和"做了给谁用"

**产出：**
- 问题陈述（一句话描述要解决什么）
- 目标用户画像
- 成功标准（怎么算做成了）

**检查点：**
- [ ] 能用一句话说清楚项目目标
- [ ] 明确了目标用户是谁
- [ ] 定义了可衡量的成功标准

**本项目示例：**
- 问题陈述：音乐制作者用 FL Studio 编曲时，大量重复操作（调 Mixer、写鼓组、整理工程）可以通过 AI Agent 自动化
- 目标用户：FL Studio 用户，有编曲经验，习惯 Piano Roll 操作
- 成功标准：用自然语言描述意图，AI 能在 FL Studio 里执行操作并返回结果

---

## 阶段二：调研（Feasibility Study）

**目标：** 搞清楚"能不能做"和"别人怎么做的"

**子步骤：**
1. **技术调研** — 相关技术栈、协议、API 能力
2. **竞品调研** — 已有方案的功能、架构、局限
3. **生态调研** — 官方动向、第三方工具、社区状态
4. **资源评估** — 时间、人力、技术门槛、成本

**产出：**
- 技术可行性报告
- 竞品对比分析
- 风险清单

**检查点：**
- [ ] 确认技术上可行（有 API / 有协议 / 有路径）
- [ ] 了解了现有方案的优劣
- [ ] 识别了关键技术风险
- [ ] 评估了资源投入是否可承受

**本项目状态：** ✅ 已完成（见 `docs/research/`）

---

## 阶段三：需求定义（Requirements Definition）

**目标：** 明确"做什么"和"做到什么程度"

**子步骤：**
1. **需求获取** — 从用户视角列出所有想要的能力
2. **需求分析** — 分类、去重、识别依赖关系
3. **需求规格** — 写成可执行的功能清单（不是模糊的"支持 XXX"）
4. **需求验证** — 确认需求是对的、完整的、可实现的

**产出：**
- 功能清单（Feature List）— 每个功能有：描述、优先级、验收标准
- 非功能需求（性能、兼容性、安全等）
- 需求优先级矩阵（MoSCoW：Must / Should / Could / Won't）

**检查点：**
- [ ] 每个需求都有明确的验收标准
- [ ] 需求按优先级排序
- [ ] 没有互相矛盾的需求
- [ ] 需求范围可控（MVP 不贪多）

**本项目示例（见 `docs/requirements/`）：**
- 场景分析已完成
- 需要补充：功能清单 + 优先级 + 验收标准

---

## 阶段四：架构设计（System Design）

**目标：** 明确"怎么组织"和"用什么技术"

**子步骤：**
1. **架构设计** — 模块划分、数据流、接口定义
2. **技术选型** — 语言、框架、依赖库、部署方式
3. **详细设计** — 每个模块的输入/输出/处理逻辑

**产出：**
- 架构图（模块关系 + 数据流）
- 技术选型文档
- 模块接口定义
- 目录结构设计

**检查点：**
- [ ] 模块边界清晰，职责单一
- [ ] 技术选型有依据（不是拍脑袋）
- [ ] 接口定义可测试
- [ ] 考虑了扩展性（未来加功能不推翻重来）

**本项目示例：**
- MCP Server 模块划分待设计
- 通信层（SysEx vs 文件队列 vs 混合）待确定
- 需要画架构图

---

## 阶段五：技术原型（Spike / Prototype）

**目标：** 用最小成本验证关键技术点

**原则：**
- 只验证**不确定**的部分，已知的不用验证
- 代码量控制在 50~200 行
- 验证完就可以扔掉，不要舍不得

**产出：**
- 原型代码（可运行，能证明可行性）
- 验证结论（能/不能，有什么坑）

**检查点：**
- [ ] 关键技术风险已验证
- [ ] 确认了通信链路能跑通
- [ ] 记录了遇到的问题和解决方案
- [ ] 决定是否继续推进

**本项目需要验证的点：**
- [ ] MCP Server 能否通过 SysEx 跟 FL Studio 通信
- [ ] Piano Roll 脚本能否被外部触发并写入音符
- [ ] mimo 模型能否正确理解音乐术语并生成有效指令

---

## 阶段六：项目计划（Project Planning）

**目标：** 明确"什么时候做什么"

**产出：**
- 里程碑定义（Phase 0/1/2/...）
- 每个里程碑的功能范围和时间估算
- 任务拆分（可执行的 Todo List）
- 风险应对计划

**检查点：**
- [ ] 里程碑可衡量（有明确的完成标准）
- [ ] 任务粒度合适（每个任务 1~4 小时可完成）
- [ ] 考虑了缓冲时间
- [ ] 识别了阻塞依赖

**本项目示例：** 见 `planning/roadmap.md`

---

## 阶段七：迭代实施（Implementation）

**目标：** 按计划逐步交付可用功能

**每个迭代的流程：**
```
选任务 → 编码 → 测试 → 集成 → 回顾 → 选下一个任务
```

**产出：**
- 可运行的代码
- 测试用例
- 变更日志
- 迭代回顾（做了什么、遇到什么、下轮做什么）

**检查点：**
- [ ] 每个迭代有可演示的成果
- [ ] 代码有基本的测试覆盖
- [ ] 遇到的问题及时记录
- [ ] 根据实际情况调整计划（拥抱变化）

## 模型应用指南

### 个人项目的灵活度

| 阶段 | 正式团队 | 个人项目 | 灵活度 |
|------|---------|---------|--------|
| 基础诉求 | PRD 文档 | 一段话 | 可轻量 |
| 调研 | 竞品分析报告 | 笔记 + 链接 | 可轻量 |
| 需求定义 | SRS 文档 | 功能清单 + 优先级 | **不能省** |
| 架构设计 | 架构评审 | 架构图 + 模块说明 | **不能省** |
| 技术原型 | Spike 报告 | 50 行代码验证 | **强烈建议** |
| 项目计划 | 甘特图 | Roadmap + Todo | 可轻量 |
| 迭代实施 | Sprint + 站会 | 按里程碑推进 | 可轻量 |

### 不能省的三件事

1. **需求定义** — 不写清楚就会做偏，个人项目最常见的失败原因
2. **架构设计** — 不设计就会写成一坨，后期改不动
3. **技术原型** — 不验证就会在实施阶段踩大坑
